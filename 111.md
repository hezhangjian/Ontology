# Dataset-first 通用平台改造方案

> 日期：2026-07-21
>
> 状态：待实施
>
> 目标：以 Dataset 为数据处理和复用中心，让普通用户按“连接数据 → 生成数据集 → 创建业务对象 → 制作看板”的单一主路径完成工作。

## 1. 产品决策

### 1.1 唯一主路径

```text
上传文件或连接数据源
  → Pipeline 清洗、转换、去重和关联
  → 生成 Dataset
  → 预览和确认 Dataset
  → 从 Dataset 创建业务对象
  → 自动映射字段
  → 建立业务关系
  → 使用 Dataset 或业务对象制作看板
```

Pipeline 的默认产物是 Dataset，不再要求用户事先建立本体，也不在默认 DAG 中放置 `ONTOLOGY_OBJECT` 或 `ONTOLOGY_RELATION` 节点。

Palantir Pipeline Builder 同时支持 Dataset 和 Ontology 输出，Ontology Manager 也支持选择 backing datasource 后自动将数据列映射为对象属性。本平台保留这个能力边界，但普通用户只看到 Dataset-first 主路径。

### 1.2 为什么不让 Pipeline 默认直接生成本体

一份清洗后的 Dataset 可以同时服务于：

- 看板和临时分析；
- 多个业务对象；
- 多条业务关系；
- 后续 Pipeline 的输入；
- 导出和其他应用。

如果把业务对象节点直接塞进 Pipeline，同一份数据会反复配置字段、主键和关系，DAG 也会混合“数据处理”与“业务建模”两种心智模型。先产出 Dataset，再从 Dataset 建模，更容易理解和复用。

### 1.3 用户不看技术治理概念

普通用户界面不出现修订、差异、提议、审批、技术标识或并发控制字段。底层为安全、幂等和故障恢复保留的技术数据，不得进入普通用户菜单、页面和操作流程。

## 2. Dataset 核心模型

### 2.1 定义

Dataset 是一组有明确字段、数据类型、来源和可查询内容的数据产物。它是 Pipeline 的标准输出，也是业务对象和看板的标准输入。

Dataset 页面仅向普通用户展示：

- 数据预览；
- 字段名称和类型；
- 来源 Pipeline；
- 行数和最近运行状态；
- “创建业务对象”和“制作看板”操作。

### 2.2 存储边界

```text
Dataset 正文
  → MinIO 中的 Parquet 文件

Dataset 名称、字段、来源和运行状态
  → PostgreSQL

Dataset 的筛选和看板查询副本
  → OpenSearch
```

OpenSearch 中的 Dataset 内容只是查询副本，可从 MinIO 中的 Dataset 重新生成。OpenSearch 不是 Dataset 的唯一存储位置。

### 2.3 数据集列表与详情

新增页面：

```text
/data/datasets
/data/datasets/:datasetId
```

列表页提供搜索、来源 Pipeline、行数和状态。详情页提供数据预览、字段列表、来源信息和下一步操作。

## 3. Pipeline Builder 改造

### 3.1 默认 DAG

新建 Pipeline 后的默认图为：

```text
SOURCE → SELECT → DATASET_OUTPUT
```

`DATASET_OUTPUT` 只需要配置：

- Dataset 名称；
- Dataset 描述；
- 输出字段。

用户不需要选择业务对象、对象主键或属性映射。

### 3.2 本地文件导入

用户界面统一为“本地文件”，支持：

- `.csv`；
- `.xls`；
- `.xlsx`。

平台自动完成：

1. 字符编码和表头检查；
2. 字段名清理；
3. 日期、数字、布尔和文本推断；
4. Excel sheet 识别，每个 sheet 作为一个资产；
5. 样例值和空值情况预览。

底层仍可将上传文件放入 MinIO 并按 S3 资产读取，不需要为用户暴露 `S3_CSV` 等技术类型。

### 3.3 运行完成页

Pipeline 成功后显示：

```text
数据集已生成

名称：员工 Token 明细
行数：6000
字段：12

[查看数据集]  [创建业务对象]  [制作看板]
```

“创建业务对象”只是跳转到业务建模向导，并自动带入当前 Dataset；它不在 Pipeline DAG 里增加本体节点。

### 3.4 高级能力边界

平台后续可支持从 Pipeline 直接发布业务对象，但必须满足：

1. 普通用户默认不显示；
2. 先物化为 Dataset，不得跳过 Dataset；
3. 使用与“从 Dataset 创建业务对象”相同的映射服务；
4. 不得让两条路径产生不同结果。

## 4. 本体管理与建模流程

### 4.1 导航与删除范围

本体管理使用稳定、可预期的导航：

```text
本体管理
├── 概览
├── 对象类型
├── 关系类型
├── 属性
├── 高级能力
│   ├── 接口
│   ├── 动作
│   └── 函数
└── 设置
```

“高级能力”默认折叠，不影响普通用户建立对象、属性和关系。Interface、Action 和 Function 在页面中统一使用“接口”“动作”“函数”。

从用户产品中完全删除：

- 变更历史；
- 变更提议；
- 健康问题。

同时删除它们在 `OntologyResourceLayout.tsx`、`App.tsx`、面包屑和 Portal API 封装中的菜单、路由和入口。底层为了幂等、恢复和审计保留的技术记录不进入本体管理界面。

### 4.2 概览页是建模入口

本体为空时，概览页只展示两个入口：

```text
从数据集创建对象（推荐）
手工创建空对象
```

已有对象后，概览页显示可缩放的“对象—关系图”。对象节点显示名称、图标、属性数和数据量；关系边显示双向名称和数量关系。点击节点或边可直接进入详情。

页面右上角提供统一“新建”下拉菜单：

```text
新建
├── 对象类型
├── 关系类型
├── 接口
├── 动作
└── 函数
```

对象详情内的快捷入口会自动带入当前对象，例如“为员工创建关系”或“为员工创建动作”。

当前项目空间内不需要先创建一个空的“本体容器”。对象类型、对象属性、关系类型以及围绕对象建立的能力共同组成业务模型；创建第一个对象类型，就是开始建立本体。

普通用户可按下面的判断选择要创建的内容：

| 用户想表达的内容 | 应创建的内容 | 例子 |
|---|---|---|
| 一类可被识别和查询的业务实体 | 对象类型 | 员工、设备、订单 |
| 对象自身的一项数据 | 属性 | 员工姓名、设备状态、订单金额 |
| 两类对象之间的业务联系 | 关系类型 | 员工属于部门、订单包含商品 |
| 多类对象都应提供的共同字段能力 | 接口 | 员工和客户都实现“人员”接口 |
| 用户可以执行并写入数据的操作 | 动作 | 调整部门、关闭订单 |
| 可重复使用的只读查询或计算 | 函数 | 查找异常设备、计算风险分 |

首次建模只引导“对象类型 → 属性 → 关系类型”。接口、动作和函数在真实需要出现后再从“高级能力”中创建。

### 4.3 创建对象类型

对象创建向导提供两种起点：

1. 从 Dataset 创建：默认路径，自动读取字段和样例数据；
2. 手工创建：用于尚无外部数据、准备由动作或 API 写入的对象，系统为它准备受管空 Dataset。

从 Dataset 创建时使用以下步骤：

#### 步骤一：选择数据

- 选择 Dataset；
- 查看字段、行数和样例数据；
- 可从 Dataset 详情页进入并自动带入当前 Dataset。

#### 步骤二：定义对象

- 输入单数名称和复数名称；
- 选择图标和颜色；
- 输入说明；
- 技术名由系统生成，仅在“高级”中可查看。

#### 步骤三：选择对象生成方式

| 生成方式 | 适用情况 | 处理 |
|---|---|---|
| 每行一个对象 | 所选标识在 Dataset 中已唯一 | 直接映射 |
| 按标识合并 | 同一对象在 Dataset 中重复出现 | 选择属性后按标识去重 |
| 层级字段展开 | 多个字段表示同一种层级对象 | 展开为对象行并生成父子关系 |

平台根据字段重复率和名称特征提供推荐，但最终选择由用户确认。

#### 步骤四：映射属性

字段映射表包含：

| 配置 | 默认行为 |
|---|---|
| Dataset 字段 | 自动填充 |
| 属性名称 | 根据字段名预填，可修改 |
| 数据类型 | 根据 Schema 预选，可修正 |
| 包含 | 默认包含，可取消 |
| 必填 | 根据空值情况建议 |
| 敏感 | 默认关闭，用户可标记 |
| 搜索、筛选、排序 | 按类型生成合理默认值 |

支持的基础类型至少包含布尔、日期、日期时间、小数、整数和文本。关系不作为普通属性类型，而在关系向导中建立。

#### 步骤五：确认身份和预览

- 选择一个唯一标识；
- 选择一个显示名称；
- 预览“Dataset 行数 → 预计对象数”；
- 显示空标识、重复标识和属性冲突；
- 展示 10—20 个对象样例。

只有唯一标识完整、稳定且没有属性冲突时才能创建。创建成功后直接进入对象详情，不再跳转到中间治理流程。

### 4.4 管理属性

属性主要在对象详情中管理。“属性”菜单提供跨对象搜索和查看，不提供脱离对象的“新建属性”按钮。

对象详情中可以：

- 从当前 Dataset 添加尚未映射的字段；
- 手工添加由动作或 API 填充的属性；
- 修改名称、说明、必填、敏感、搜索、筛选和排序设置；
- 预览属性样例值和空值占比；
- 在删除前查看看板、关系、动作和函数对它的引用。

### 4.5 创建关系类型

关系可从“新建”菜单、关系类型列表或对象详情的关系图进入。向导包含：

#### 步骤一：选择两端对象

- 选择源对象和目标对象；
- 允许选择同一种对象，例如“员工汇报给员工”；
- 选择一对一、一对多、多对一或多对多。

#### 步骤二：定义双向名称

一条关系同时定义两个方向的页面名称，例如：

```text
员工 → 组长：汇报给
组长 → 员工：管理成员
```

不为反向查询再创建第二条关系类型。

#### 步骤三：选择关系数据来源

| 来源方式 | 适用关系 | 配置 |
|---|---|---|
| 外键字段 | 一对一或多对一 | 选择一端的外键属性和另一端的唯一标识 |
| Dataset 字段对 | 任意数量关系 | 选择 Dataset 中对应两端标识的字段 |
| 关联对象 | 关系本身还有重要属性 | 选择一种中间对象及其两条已有关系 |

本平台的 Dataset-first 主路径默认选择“Dataset 字段对”，因为它可以从同一个主 Dataset 中生成多种关系。

#### 步骤四：预览与创建

- 显示预计关系数；
- 显示两端成功匹配数、未匹配数和重复对数；
- 展示 10—20 条关系样例；
- 未匹配数据可返回 Dataset 查看；
- 确认后直接创建。

### 4.6 创建接口

接口用于让多种对象以同一组属性被统一查询，例如让员工、外包人员和客户都实现“人员”接口。

创建向导：

1. 输入接口名称和说明；
2. 定义必须提供的属性槽位，例如姓名、编号和状态；
3. 选择实现该接口的对象类型；
4. 将每个槽位映射到对象属性；
5. 校验类型兼容性后创建。

接口不是演示场景的必选项，不在首次建模引导中展示。

### 4.7 创建动作

动作表达用户被允许执行的业务操作，例如“调整员工所属部门”“关闭订单”或“将设备标记为离线”。用户不直接修改 HugeGraph 或 Dataset。

创建向导：

1. 选择目标对象；
2. 选择创建对象、修改属性、停用对象、建立关系或删除关系；
3. 定义用户需要填写的参数；
4. 将参数映射到属性或关系两端；
5. 设置执行权限、确认提示和可执行条件；
6. 使用样例对象预览执行前后的差异；
7. 测试通过后创建。

动作详情保留“概览、参数、规则、条件、测试、使用位置、设置”，不出现提议或中间发布页面。

### 4.8 创建函数

函数用于只读查询和可复用计算，例如“查找某部门的异常员工”或“计算设备风险分”。

创建向导：

1. 选择函数的主要对象或接口；
2. 定义输入参数；
3. 用筛选、关系遍历、分组和聚合组合只读逻辑；
4. 选择标量、布尔、对象集或表格输出；
5. 设置超时、结果上限和缓存；
6. 输入样例参数测试结果后创建。

函数详情保留“概览、输入输出、逻辑、测试、配置、使用位置、设置”。

### 4.9 对象详情是建模中心

对象详情不展示存储引擎、物理键、索引别名或技术状态。页面包含：

| 页签 | 内容 |
|---|---|
| 概览 | 名称、图标、说明、数据量和样例对象 |
| 属性 | 属性映射、数据类型、唯一标识和显示名称 |
| 数据来源 | Dataset、对象生成方式、预览和重新读取 |
| 关系 | 以当前对象为中心的关系图和快捷创建 |
| 动作 | 可对当前对象执行的业务操作 |
| 函数 | 使用当前对象的只读计算 |
| 使用位置 | 引用当前对象的关系、看板、应用和 API |
| 设置 | 负责人、标签和删除保护 |

属性、关系、动作和函数修改后直接保存。删除前必须检查使用位置，存在直接依赖时禁止删除并给出可理解的处理建议。

### 4.10 简化后的后端与 API

Portal 每次新建或编辑时只调用一次对应的保存端点，不再串联内部治理接口。公共 OpenAPI 提供面向用户操作的原子端点：

- 创建、更新和删除动作；
- 创建、更新和删除对象类型；
- 创建、更新和删除函数；
- 创建、更新和删除关系类型；
- 创建、更新和删除接口；
- 从 Dataset 预览对象映射；
- 从 Dataset 预览关系映射。

后端在一次请求中完成校验、保存和激活，任何步骤失败都返回可操作的错误信息。现有底层表和恢复机制可暂时保留为内部实现，但不得继续泄露到 Portal 或面向用户的 API 流程。

## 5. 示例 CSV 的 Dataset 和业务模型

### 5.1 输入数据

`/Users/dijkstra/project/06-shixi/data` 当前包含：

| 文件 | 数据行 | 主要字段 |
|---|---:|---|
| `demo-employee-leaders.csv` | 100 | 姓名、工号、组长姓名 |
| `demo-token-usage.csv` | 6000 | 时间、工号、Token、姓名、人员类型、编码、五层组织、Agent 类型 |

`demo-token-usage.csv` 包含 100 名员工、20 个月份和 3 种 Agent 类型，可同时支撑人员、组织、Agent 和 Token 明细建模。`demo-employee-leaders.csv` 可完整匹配出 8 个组长小组，每组 12—13 人，且每位组长本人都包含在自己的小组中。

### 5.2 Pipeline 只输出一个主 Dataset

两个 CSV 作为 Pipeline 输入，经过关联后只产出一个可分析、可建模的宽 Dataset：

```text
demo-token-usage.csv
  → 清理时间、工号、Token、人员、组织和 Agent 字段
  → LEFT JOIN demo-employee-leaders.csv ON 工号
  → 根据组长姓名匹配组长工号
  → 生成 Token 明细唯一标识
  → 将组长工号同时作为组长小组标识
  → dataset_token_usage_enriched
```

这个 Dataset 保持 Token 明细粒度，预计为 6000 行，并包含看板和建模需要的全部字段：

| 字段 | 用途 |
|---|---|
| `usage_id` | 由月份、员工标识和 Agent 类型组合生成的 Token 明细唯一标识 |
| `month` | 使用月份 |
| `employee_id` | 员工唯一标识 |
| `employee_name` | 员工显示名称 |
| `employee_code` | 业务编码 |
| `employee_type` | 人员类型 |
| `leader_employee_id` | 组长的员工标识 |
| `leader_employee_name` | 组长姓名 |
| `leader_group_id` | 组长小组标识，值为组长工号 |
| `leader_group_name` | 组长小组显示名称，值为组长姓名 |
| `agent_type_id` | Agent 类型标识 |
| `agent_type_name` | Agent 类型显示名称 |
| `total_tokens` | Token 数量 |
| `level_1_org_id` 至 `level_5_org_id` | 按完整组织路径生成的各级组织标识 |
| `level_1_org_name` 至 `level_5_org_name` | 组织路径和看板维度 |

小组的定义是“同一位组长本人及其组员”。小组按组长工号区分，组长姓名只用于页面显示。

### 5.3 一个 Dataset 创建多种业务对象

用户在 `dataset_token_usage_enriched` 详情页点击“创建业务对象”，可以反复从同一个 Dataset 创建不同对象。Dataset 不会因为创建多个对象而被复制。

由于主 Dataset 的每一行是一条 Token 明细，员工、组织、小组和 Agent 都会重复出现。业务对象的唯一标识必须唯一，因此平台在后台为每种对象生成“对象投影视图”：

| 业务对象 | 源 Dataset | 对象投影规则 | 唯一标识 | 显示名称 |
|---|---|---|---|---|
| `Employee` / 员工 | `dataset_token_usage_enriched` | 选择员工字段，按工号去重 | `employee_id` | `employee_name` |
| `OrganizationUnit` / 组织单元 | `dataset_token_usage_enriched` | 将一至五级组织展开为组织行，按完整路径去重 | `org_unit_id` | `org_unit_name` |
| `LeaderGroup` / 组长小组 | `dataset_token_usage_enriched` | 选择组长工号和姓名，按组长工号去重 | `leader_group_id` | `leader_group_name` |
| `AgentType` / Agent 类型 | `dataset_token_usage_enriched` | 选择 Agent 字段，按 Agent 标识去重 | `agent_type_id` | `agent_type_name` |
| `TokenUsage` / Token 使用明细 | `dataset_token_usage_enriched` | 保留明细行 | `usage_id` | `month` |

对象投影视图是业务对象的内部读取结果，不作为新 Dataset 出现在数据集列表中。普通用户始终只看到一个主 Dataset。

如果同一个员工标识对应了多个姓名或多个组长，平台不得随意选一条，而应在创建前明确提示冲突，引导用户回到 Pipeline 清理数据。

### 5.4 从同一 Dataset 创建业务关系

| 源对象 | 关系 | 目标对象 | 源 Dataset 字段 |
|---|---|---|---|
| 员工 | 汇报给 | 员工 | `employee_id → leader_employee_id` |
| 员工 | 属于 | 组长小组 | `employee_id → leader_group_id` |
| 员工 | 属于 | 组织单元 | `employee_id → level_5_org_id` |
| 组织单元 | 隶属于 | 组织单元 | `org_unit_id → parent_org_unit_id` |
| Token 使用明细 | 由员工产生 | 员工 | `usage_id → employee_id` |
| Token 使用明细 | 属于 | 组长小组 | `usage_id → leader_group_id` |
| Token 使用明细 | 使用 Agent 类型 | Agent 类型 | `usage_id → agent_type_id` |

关系也按两端标识去重，不会因为主 Dataset 中有多个月份或多种 Agent 而重复生成同一条关系。

看板的部门、小组和月份聚合直接使用 `dataset_token_usage_enriched` 中的扁平分析字段，不要求 OpenSearch 在看板查询时跨 HugeGraph 关系补齐维度。

### 5.5 什么时候才拆分 Dataset

默认不拆分。只有出现以下需求时，才由用户或平台建议将主 Dataset 派生为其他 Dataset：

- 数据属于不同负责团队或不同可见范围；
- 数据需要不同的刷新节奏；
- 下游多处都要复用同一份独立中间结果；
- 行数、字段数或计算成本已经明显影响查询。

## 6. 通用分析看板设计

### 6.1 设计原则

看板不得把“部门”“组长”“Token”等演示概念写死在代码中。平台只提供通用的数据源、维度、指标、筛选、排序和展示能力，演示场景通过页面配置完成。

看板支持两种数据源：

1. Dataset：默认选择，适合报表、分组、趋势和临时分析；
2. 业务对象：适合对象搜索、对象列表和下钻到对象详情。

新建组件时使用统一配置流程：

```text
选择数据源
  → 选择展示方式
  → 选择行维度和列维度
  → 选择指标及计算方式
  → 配置筛选、排序和 Top N
  → 配置数值格式和下钻
```

### 6.2 通用维度与指标

维度由 Dataset 字段配置得到，支持：

- 分类维度，例如部门、产品、地区或组长小组；
- 对象引用维度，例如员工、客户或设备；
- 时间维度，可选日、周、月、季或年；
- 数值分箱维度，例如金额区间或时长区间。

指标计算支持：

| 页面名称 | 通用计算 | 例子 |
|---|---|---|
| 平均值 | `AVG(measure)` | 平均订单金额 |
| 记录数 | `COUNT(*)` | 订单数 |
| 去重数 | `DISTINCT_COUNT(field)` | 使用人数 |
| 合计 | `SUM(measure)` | Token 总量 |
| 人均、设备均等 | `SUM(measure) / DISTINCT_COUNT(entity)` | 人均 Token |
| 日均、月均等 | `SUM(measure) / DISTINCT_COUNT(period)` | 小组月均 Token |
| 最大值 | `MAX(measure)` | 单日最高用量 |
| 最小值 | `MIN(measure)` | 单日最低用量 |

“人均”和“月均”共用同一个通用能力 `SUM_PER_DISTINCT`，只是去重计数字段不同。页面将它表达为：

```text
计算：求和 ÷ 去重数
求和字段：[Token 数量]
去重字段：[员工] 或 [月份]
```

平台可提供“人均”和“期均”快捷选项，但底层不绑定人员、月份或 Token 字段。

### 6.3 表格是正式组件

看板不得只展示图。组件库必须提供三种表格：

| 类型 | 配置 | 用途 |
|---|---|---|
| 明细表 | 选择需要展示的原始字段 | 查看明细、排序、分页和导出 |
| 汇总表 | 选择多个维度和指标 | 展示按维度分组的聚合结果 |
| 透视表 | 选择行维度、列维度和指标 | 生成“部门 × 月份”等交叉表 |

透视表必须支持：

- 合计行和合计列；
- 表头固定和横向滚动；
- 空值、零值和无数据的明确区分；
- 按合计或指定月份排序；
- 数值格式、小数位和千分位；
- 单元格点击后打开对应明细；
- CSV 导出。

透视表使用 `antd` Table 渲染动态列；热力图是独立的可视化组件，不得用热力图代替透视表。

### 6.4 图表组件与技术选型

前端统一使用：

- `@ant-design/charts`：面积图、柱状图、环形图、折线图、饼图和热力图；
- `@ant-design/icons`：图标；
- `antd`：指标卡、筛选器、明细表、汇总表、对象表和透视表。

`WidgetFrame.tsx` 不再使用 `<Progress>` 模拟图表。图和表可共用同一份查询定义，用户切换展示方式时不需要重新选择维度和指标。

### 6.5 筛选器

按公共 API 契约渲染：

- `BOOLEAN`：开关；
- `DATE_RANGE`：日期范围；
- `DROPDOWN`：单选下拉；
- `MULTISELECT`：多选下拉；
- `NUMBER_RANGE`：数值范围；
- `OBJECT_REF`：业务对象选择。

选项由平台 API 返回，Portal 不直接请求 OpenSearch。看板顶部的全局筛选可同时影响图、汇总表和透视表。

### 6.6 运行时维度切换

组件可保存多组允许的维度组合，例如：

```text
一级部门 × 月份
三级部门 × 月份
五级部门 × 月份
组长小组 × 月份
Agent 类型 × 月份
```

查询请求可携带维度覆盖，后端必须确认：

1. 字段存在于当前数据源；
2. 字段允许分组；
3. 字段对当前用户可见；
4. 维度数量没有超过平台限制。

### 6.7 演示看板布局

演示看板使用 `dataset_token_usage_enriched`，但所有组件均由通用配置生成，不新增 Token 专用接口或页面。

页面顶部筛选器：

```text
时间范围  部门层级  部门  组长小组  Agent 类型  人员类型
```

第一区：部门月份 Token 消耗

| 配置项 | 值 |
|---|---|
| Dataset | `dataset_token_usage_enriched` |
| 行维度 | 用户选择的部门层级 |
| 列维度 | `month` |
| 指标 | `SUM(total_tokens)` |
| 展示 | 堆叠柱状图 + 透视表 |

第二区：部门月份人均 Token 消耗

| 配置项 | 值 |
|---|---|
| Dataset | `dataset_token_usage_enriched` |
| 行维度 | 用户选择的部门层级 |
| 列维度 | `month` |
| 分子 | `SUM(total_tokens)` |
| 分母 | `DISTINCT_COUNT(employee_id)` |
| 展示 | 折线图 + 透视表 |

透视表单元格除人均值外，详情提示同时展示 Token 总量和去重人数，避免用户无法判断人均值的分母。

第三区：组长小组 Token 月均消耗

| 配置项 | 值 |
|---|---|
| Dataset | `dataset_token_usage_enriched` |
| 行维度 | `leader_group_id`，显示 `leader_group_name` |
| 分子 | `SUM(total_tokens)` |
| 分母 | `DISTINCT_COUNT(month)` |
| 展示 | 横向柱状图 + 汇总表 |

第三区还提供“查看每月”切换：切换后以组长小组为行、月份为列，展示每个小组每月的 `SUM(total_tokens)` 透视表。

小组汇总必须遵守：

1. 以 `leader_group_id` 分组，不以组长姓名分组；
2. 组长本人和所有组员的 Token 都计入该组；
3. 同名组长仍是不同小组；
4. 无法匹配组长的员工进入“未分组”，并在 Dataset 质量提示中列出。

### 6.8 分母口径

“人均”默认使用当前筛选范围内有明细记录的去重人数。如果业务要求将零使用人员也纳入分母，Pipeline 必须先将人员花名册与时间范围生成完整的“人员 × 期间”数据，再左关联 Token 明细并将缺失用量补为零。

“月均”默认除以当前筛选范围内实际出现的去重月份数。如果要求缺失月份也计入分母，Pipeline 必须使用时间维表补齐小组与月份的组合。

两种口径都由 Dataset 和指标配置决定，看板展示指标说明，不在前端暗中改变计算结果。

## 9. 验收标准

### 9.1 Dataset 主路径

1. 用户可上传两个示例 CSV；
2. Pipeline 默认为 `SOURCE → SELECT → DATASET_OUTPUT`；
3. Pipeline 不要求事先存在业务对象；
4. Pipeline 成功后可打开 Dataset 预览；
5. Dataset 可被其他 Pipeline、业务建模和看板重复使用。

### 9.2 业务建模

1. 本体导航不再出现变更提议、健康问题和变更历史，直接输入相应旧地址也不会打开旧页面；
2. 概览页可从空状态引导建模，已有对象时显示对象—关系图；
3. 统一“新建”菜单可创建对象类型、关系类型、接口、动作和函数；
4. 用户可从 Dataset 详情点击“创建对象”，也可手工创建空对象；
5. Dataset 字段自动出现在属性映射表中，用户可确认对象生成方式、唯一标识、显示名称和属性；
6. 创建前可看到预计对象数、样例对象、空标识、重复标识和属性冲突；
7. 用户可从一个主 Dataset 建立 Employee、OrganizationUnit、LeaderGroup、TokenUsage 和 AgentType；
8. 用户可建立员工、组织、组长小组、Agent 和 Token 明细之间的双向关系；
9. 对象详情可管理属性、数据来源、关系、动作、函数和使用位置；
10. 接口、动作和函数均可独立创建、测试和保存，不跳转到中间治理页面。

### 9.3 分析看板

1. 用户可直接选择 `dataset_token_usage_enriched` 制作看板；
2. “部门月份 Token 消耗”同时以堆叠柱状图和透视表展示 `SUM(total_tokens)`；
3. “部门月份人均 Token 消耗”同时以折线图和透视表展示 `SUM(total_tokens) / DISTINCT_COUNT(employee_id)`；
4. “组长小组 Token 月均消耗”以横向柱状图和汇总表展示 `SUM(total_tokens) / DISTINCT_COUNT(month)`；
5. 用户可切换到小组每月 Token 合计透视表；
6. 组长本人和所有组员的 Token 都计入该组，小组按组长工号分组并显示组长姓名；
7. 用户可切换部门层级，也可改用 Agent 类型等其他维度；
8. 图表由 Ant Design Charts 渲染，表格由 `antd` 渲染，不再用进度条代替；
9. 筛选器可按部门、组长小组、人员、时间和 Agent 类型生效；
10. 上述结果全部通过通用维度和指标配置得到，代码中没有 Token 场景专用分支。

## 10. 风险与处理

| 风险 | 处理方式 |
|---|---|
| CSV 含 BOM 或字段名不规范 | 导入时自动清理，预览时显示结果 |
| Dataset 查询副本暂时不可用 | 不影响 MinIO 中的 Dataset 正文，恢复后重新生成 |
| Excel 内容过大 | 限制单文件和总上传大小，超限时提示分拆 |
| 组长姓名不唯一 | 要求匹配到组长工号，冲突记录进入质量错误列表 |
| 类型推断不准 | 在 Dataset 预览和对象映射向导中允许修正 |
| 组织同名 | 使用完整组织路径生成稳定标识 |
